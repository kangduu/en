# 贡献指南（Contributing）

欢迎加入本项目！感谢你愿意贡献。  
请在提交代码、报告问题或提出功能建议前，先阅读本指南，以便大家更高效、愉快地协作。

## 一、我们是谁

本仓库由 **团队 [kangduu]** 维护，目标是 … 。

## 二、如何贡献

你可以通过以下几种方式参与：

- 提交 Bug 报告（Issue）
- 提出功能建议（Feature Request）
- 编写或改进代码（Pull Request）
- 改进文档、示例或测试
- 提供其它类型的贡献（例如：UI 设计、用户反馈、翻译等）

## 三、报告问题

当你发现 bug 或问题时，请：

1. 搜索现有的 Issue，看是否已有相同报告。
2. 如果没有，则创建一个新 Issue，描述清晰包括：
   - 问题发生的环境（系统／浏览器／版本等）
   - 重现步骤
   - 期望行为 vs 实际行为
   - 如果方便，提供最小可复现示例或截图
3. 在 Issue 标题中简明说明问题，例如 “Bug: 某 xx 功能在 … 时崩溃”。

## 四、提出新功能／改进

如果你希望新增功能或改进现有功能，请：

- 优先在 Issue 中讨论，以确认该功能是否符合项目方向。
- 提供功能描述、使用场景、设计草案（可选）、影响范围。
- 讨论达成一致后，再新建 Branch 并发起 Pull Request。

## 五、Pull Request 流程

1. 从 `develop` 分支（或指定分支）切出你的 feature 分支，命名建议：`feature/xxx` 或 `bugfix/xxx`。
2. 在本地完成开发／测试，包括：
   - 代码风格符合项目规范
   - 单元测试、集成测试已通过（如有）
   - 文档（README/CHANGELOG）已更新（如有必要）
3. 将 Branch 推送到远程仓库，并发起 Pull Request（PR）指向 `develop`。
4. 审核流程：至少 1 位团队成员审查，CI 测试通过后由负责人员合并。
5. 合并后删除 feature 分支。
6. 若为发布上线，请依据发布流程合并至 `main` 分支。

## 六、代码规范 &测试

- 请遵循项目的代码风格。
- 在提交前请执行本地测试：
  ```bash
  npm test
  ```
- 若改动较大，请考虑如何覆盖单元或集成测试。
- 所有 console.log 调试语句请移除，提交前请检查。

## 七、感谢你的贡献

无论你提交的是 bug 报告、文档改善、还是完整的功能，我们都非常感激你的参与！希望你在这里有愉快的协作体验。
